# Tema 9. COPILOT APLICADO A SPRING BOOT: PRODUCTIVIDAD SIN ROMPER ESTANDARES

En este tema aterrizamos todo lo visto hasta ahora en un flujo real de construccion de microservicios con Spring Boot. La meta no es "generar codigo rapido", sino entregar codigo que compila, pasa tests, respeta arquitectura y se mantiene en produccion.

Aunque el titulo mantiene la referencia historica a Copilot, el enfoque practico de este tema es **agente multi-modelo**: puedes ejecutar el mismo flujo con Copilot Agent, Claude Code u otras herramientas equivalentes, seleccionando el modelo por tier segun el riesgo tecnico de la tarea.

Marco operativo del Senior:

- La IA ejecuta.
- Tu defines especificacion, limites y criterios de calidad.
- Ningun cambio entra a `main` sin revision tecnica.

Stack de referencia para este tema (estandares modernos):

- Java 21+
- Spring Boot 3.4+
- Maven 3.9+
- OpenAPI 3.1
- Testcontainers 1.19+
- JUnit 5
- Docker (estable)

Politica de tier por tipo de trabajo:

- Tier alto (razonamiento profundo): arquitectura, ADR, contratos complejos.
- Tier balanceado: implementacion multiarchivo, code review, integracion.
- Tier rapido: boilerplate, mappers, tests unitarios simples.

## 1. Scaffolding controlado: la estructura la define el Senior

Un agente multi-modelo acelera mucho en scaffolding, pero si no acotas la mision te mezcla capas, rompe naming y contamina dominio con detalles de infraestructura.

### Objetivo del scaffolding en Spring

Generar rapido el esqueleto de una feature respetando:

- Arquitectura del proyecto (hexagonal o por capas).
- Convenciones de paquetes y nombres.
- Contrato API existente (OpenAPI si aplica).
- Java moderno (records para DTOs, inyeccion por constructor).

### Antes de pedir codigo: define el contrato minimo

No arranques por `@RestController`. Arranca por especificacion.

Contrato minimo recomendado:

1. OpenAPI 3.1 (si hay API HTTP).
2. Definicion de puertos de entrada/salida.
3. Reglas de dominio no negociables.
4. Criterios de aceptacion verificables.

Ejemplo de contrato OpenAPI minimo para scaffolding:

```yaml
openapi: 3.1.0
info:
  title: Inventory API
  version: 1.0.0
paths:
  /api/inventory:
    post:
      operationId: createInventoryItem
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreateInventoryItemRequest'
      responses:
        '201':
          description: Created
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/InventoryItemResponse'
components:
  schemas:
    CreateInventoryItemRequest:
      type: object
      required: [sku, name, quantity]
      properties:
        sku:
          type: string
        name:
          type: string
        quantity:
          type: integer
          minimum: 0
    InventoryItemResponse:
      type: object
      properties:
        id:
          type: string
          format: uuid
        sku:
          type: string
        name:
          type: string
        quantity:
          type: integer
```

### Prompt al agente (plantilla reusable)

> "Crea la feature `Inventory` en Spring Boot 3.4 con Java 21. Respeta arquitectura hexagonal existente.\n> Archivos esperados:\n> - `domain/model/InventoryItem.java`\n> - `domain/port/in/CreateInventoryItemUseCase.java`\n> - `domain/port/out/InventoryRepository.java`\n> - `domain/service/InventoryService.java`\n> - `infrastructure/rest/InventoryController.java`\n> - `infrastructure/persistence/InventoryJpaAdapter.java`\n> Restricciones:\n> - No uses `@Autowired` en campos\n> - DTOs como `record`\n> - No pongas anotaciones JPA en `domain/`\n> Criterio de exito:\n> - `mvn test` en verde\n> - Sin imports de `infrastructure` dentro de `domain`."

### Prompt mejorado con enfoque multi-modelo

> "Implementa scaffolding de `Inventory` con Java 21 y Spring Boot 3.4.\n> Usa OpenAPI `openapi/inventory-api.yaml` como fuente de verdad.\n> Si falta informacion, pregunta antes de generar codigo.\n> Devuelveme primero un plan de archivos (create/modify/read-only).\n> Solo despues de mi confirmacion aplica cambios.\n> Ejecuta `mvn -q test` al final y reporta resultado."

### Ejemplo de borde correcto (Controller delgado)

```java
@RestController
@RequestMapping("/api/inventory")
public class InventoryController {

    private final CreateInventoryItemUseCase createInventoryItemUseCase;

    public InventoryController(CreateInventoryItemUseCase createInventoryItemUseCase) {
        this.createInventoryItemUseCase = createInventoryItemUseCase;
    }

    @PostMapping
    public ResponseEntity<InventoryItemResponse> create(@Valid @RequestBody CreateInventoryItemRequest request) {
        InventoryItem created = createInventoryItemUseCase.create(
            new CreateInventoryItemCommand(request.sku(), request.name(), request.quantity())
        );
        return ResponseEntity.status(HttpStatus.CREATED).body(InventoryItemResponse.from(created));
    }
}
```

### Anti-patrones que debes bloquear desde el prompt

- Meter logica de negocio en `@RestController`.
- Repositorios JPA expuestos directamente al controlador.
- DTO mutable con Lombok `@Data` en APIs publicas.
- `findAll()` sin paginacion para endpoints list.

### Control de calidad rapido post-scaffolding

Checklist express en 2 minutos:

- `domain/` sin dependencias de frameworks.
- adaptadores `rest/` y `persistence/` desacoplados del dominio por puertos.
- inyeccion por constructor en todas las clases Spring.
- convencion de nombres consistente (Request/Response/Command).

## 2. Manejo profesional de errores y observabilidad

Un servicio "funcional" sin estrategia de errores ni trazabilidad no es produccion-ready.

### Estandar minimo en Spring Boot 3.x

- `@ControllerAdvice` centralizado.
- `ProblemDetail` (RFC 7807) para respuestas de error.
- Correlation ID en logs (`MDC`) para trazas distribuidas.

### Prompt al agente

> "Implementa manejo global de errores para `orders-service` con `@ControllerAdvice` y `ProblemDetail` en Spring 6.\n> Incluye mapeo para:\n> - `OrderNotFoundException` -> 404\n> - `ValidationException` -> 400\n> - error inesperado -> 500 sin filtrar stacktrace al cliente\n> Agrega `correlationId` en la respuesta y en logs usando MDC.\n> No uses `e.printStackTrace()`."

### Ejemplo base con `ProblemDetail`

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(OrderNotFoundException.class)
    public ResponseEntity<ProblemDetail> handleOrderNotFound(OrderNotFoundException ex) {
        ProblemDetail problem = ProblemDetail.forStatus(HttpStatus.NOT_FOUND);
        problem.setTitle("Order not found");
        problem.setDetail(ex.getMessage());
        problem.setProperty("correlationId", MDC.get("correlationId"));
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(problem);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ProblemDetail> handleUnexpected(Exception ex) {
        ProblemDetail problem = ProblemDetail.forStatus(HttpStatus.INTERNAL_SERVER_ERROR);
        problem.setTitle("Unexpected error");
        problem.setDetail("An unexpected error occurred");
        problem.setProperty("correlationId", MDC.get("correlationId"));
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(problem);
    }
}
```

### Taxonomia de errores recomendada

No dejes que el agente improvise codigos por endpoint.

Categorias:

- `VALIDATION_ERROR` (400)
- `NOT_FOUND` (404)
- `CONFLICT` (409)
- `UNAUTHORIZED` (401)
- `FORBIDDEN` (403)
- `INTERNAL_ERROR` (500)

Ejemplo de normalizacion:

```java
public final class ErrorCodes {
    public static final String VALIDATION_ERROR = "VALIDATION_ERROR";
    public static final String NOT_FOUND = "NOT_FOUND";
    public static final String CONFLICT = "CONFLICT";
    public static final String INTERNAL_ERROR = "INTERNAL_ERROR";

    private ErrorCodes() {}
}
```

### Observabilidad minima para microservicios

- Log estructurado JSON en entornos no locales.
- `traceId` y `spanId` cuando hay trazas distribuidas.
- `correlationId` para operaciones de negocio.
- Metricas de error por endpoint y por tipo de excepcion.

Prompt orientado a observabilidad:

> "AĂ±ade filtros HTTP para generar/propagar `correlationId`.\n> Si llega header `X-Correlation-Id`, reutilizarlo; si no existe, crearlo.\n> Incluirlo en respuesta y en logs MDC.\n> AĂ±adir test de integracion para verificar propagacion."

### Checklist de observabilidad

- Logs con contexto de negocio (`orderId`, `customerId`, `correlationId`).
- Sin datos sensibles (password, tokens, PAN, PII).
- Sin logs duplicados ruido en bucles calientes.
- Cada error de negocio tiene codigo y mensaje consistente.

### Anti-patrones frecuentes generados por agentes

- `catch (Exception e)` en controladores para todo.
- devolver stacktrace al cliente por "debug".
- mezclar errores tecnicos y de dominio sin taxonomia.
- loggear payload completo con informacion sensible.

## 3. Persistencia: rendimiento, transacciones y concurrencia

La IA suele generar persistencia "que pasa tests locales" pero no escala. Aqui es donde un Senior evita deuda tecnica temprana.

### 3.1 Queries y N+1

Si el agente propone `findAll()` + filtros en Java, para y corrige. El filtrado, paginacion y joins utiles deben vivir en BD.

```java
public interface OrderJpaRepository extends JpaRepository<OrderEntity, UUID> {

    @EntityGraph(attributePaths = {"items", "customer"})
    Page<OrderEntity> findByStatus(OrderStatus status, Pageable pageable);

    @Query("""
        select o from OrderEntity o
        where o.customerId = :customerId
          and o.createdAt between :start and :end
        order by o.createdAt desc
    """)
    List<OrderEntity> findRecentByCustomer(
        @Param("customerId") UUID customerId,
        @Param("start") LocalDateTime start,
        @Param("end") LocalDateTime end
    );
}
```

Mejoras recomendadas para endurecer lo generado:

- Page en endpoints list, nunca listas infinitas.
- indices en columnas de filtro real (`customer_id`, `status`, `created_at`).
- projections cuando no necesitas entidad completa.
- evitar `JOIN FETCH` en cadena sin analizar cardinalidad.

### 3.2 Transaccionalidad real

`@Transactional` no resuelve todo. Debes separar:

- Operaciones puramente de BD (transaccion local).
- Publicacion de eventos / HTTP externo (outbox, retries, idempotencia).

Patron recomendado en operaciones criticas:

1. validar negocio.
2. persistir estado y evento en outbox en la misma transaccion.
3. publicar fuera de la transaccion con reintentos.

Ejemplo simplificado de registro outbox:

```java
@Transactional
public Order createOrder(CreateOrderCommand cmd) {
    Order order = orderFactory.create(cmd);
    Order saved = orderRepository.save(order);

    OutboxEvent event = OutboxEvent.orderCreated(saved.getId(), saved.getCustomerId());
    outboxRepository.save(event);

    return saved;
}
```

### 3.3 Concurrencia (stock, saldo, cupos)

Pide explicitamente estrategia de locking cuando el dominio lo exija.

```java
@Entity
@Table(name = "inventory")
public class InventoryEntity {

    @Id
    private UUID id;

    @Version
    private Long version;

    @Column(nullable = false)
    private Integer available;
}
```

### Prompt al agente

> "En el flujo de reserva de stock, implementa control de concurrencia con `@Version` (optimistic locking).\n> Si hay conflicto, captura `OptimisticLockException` y aplica reintento maximo 3 veces.\n> Agrega test concurrente que valide que no hay stock negativo."

### Variante cuando hay alta contencion

Cuando el mismo recurso recibe muchas escrituras simultaneas, pide alternativa explicitamente:

> "Compara optimistic locking vs `PESSIMISTIC_WRITE` vs update atomico SQL para `reserveStock`.\n> Devuelve tabla de trade-offs (latencia, throughput, complejidad, riesgo de deadlock) y recomienda opcion para 1000 req/min por SKU."

Ejemplo de update atomico:

```java
@Modifying
@Query("""
    update InventoryEntity i
       set i.available = i.available - :qty
     where i.id = :id
       and i.available >= :qty
""")
int reserve(@Param("id") UUID id, @Param("qty") int qty);
```

### Pruebas minimas obligatorias en persistencia

- test repositorio con Testcontainers (PostgreSQL 16).
- test de concurrencia para stock/saldo.
- test de performance basico para query principal (sin micro-optimizar, pero validando no N+1).

Ejemplo de base para Testcontainers:

```java
@Testcontainers
@DataJpaTest
class OrderJpaRepositoryIT {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16");

    @DynamicPropertySource
    static void props(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }
}
```

### Anti-patrones de persistencia que debes rechazar en review

- `findAll()` + stream + filtros para datos grandes.
- `@Transactional` en controladores.
- llamada HTTP externa dentro de transaccion larga.
- modificar tests para ocultar problemas de concurrencia.

## 4. Configuracion, perfiles y secret management

Un tema tipico de mala generacion IA: dejar credenciales hardcodeadas en `application.yml` o usar `@Value` de forma caotica.

### Estandar de configuracion recomendado

- `application.yml` minimo y neutral.
- `application-dev.yml`, `application-test.yml`, `application-prod.yml` por entorno.
- `@ConfigurationProperties` + validacion (`jakarta.validation`).
- Secretos por variables de entorno o gestor dedicado.

```java
@ConfigurationProperties(prefix = "payment.gateway")
@Validated
public record PaymentGatewayProperties(
    @NotBlank String baseUrl,
    @NotBlank String apiKey,
    @DurationMin(seconds = 1) Duration timeout
) {}
```

```yaml
# application-prod.yml
payment:
  gateway:
    base-url: ${PAYMENT_GATEWAY_BASE_URL}
    api-key: ${PAYMENT_GATEWAY_API_KEY}
    timeout: 2s
```

### Prompt al agente

> "Refactoriza configuraciones de `@Value` a `@ConfigurationProperties` para el modulo de pagos.\n> No dejes secretos hardcodeados.\n> Agrega validacion para que la app falle rapido al arrancar si falta `PAYMENT_GATEWAY_API_KEY`."

### Jerarquia de perfiles sugerida

- `default`: solo valores neutros.
- `dev`: comodidad local (H2 opcional, logs detallados).
- `test`: determinista para CI.
- `prod`: estrictamente por env vars/secret manager.

### Politica de secretos para equipos

- nunca commitear secretos reales.
- rotacion obligatoria de tokens expuestos.
- escaneo de secretos en pre-commit y CI (`gitleaks` o equivalente).
- separar `config` (no sensible) de `secret` (sensible).

Prompt de hardening de secretos:

> "Revisa `application*.yml` y reemplaza cualquier valor sensible por placeholders de env var.\n> Genera lista de variables requeridas por entorno (`dev`, `test`, `prod`) y valida en arranque que no falten en `prod`."

### Config fail-fast recomendado

```java
@Configuration
@EnableConfigurationProperties(PaymentGatewayProperties.class)
public class PaymentConfig {

    @Bean
    RestClient paymentRestClient(PaymentGatewayProperties props) {
        return RestClient.builder()
            .baseUrl(props.baseUrl())
            .requestFactory(new JdkClientHttpRequestFactory())
            .build();
    }
}
```

Si falta `baseUrl` o `apiKey`, la app debe fallar en bootstrap, no en runtime.

### Anti-patron critico

- Aceptar PR con secretos en texto plano porque "es solo para dev".

### Otros anti-patrones comunes

- usar `@Value` disperso en 20 clases.
- duplicar claves de config en varios `yml` sin control.
- perfiles que cambian comportamiento de negocio (no solo infraestructura).

## 5. Dockerizacion y entorno local reproducible

Si la IA te genera un `Dockerfile` que funciona solo en tu maquina, no te sirve.

### Dockerfile base recomendado (multi-stage)

```dockerfile
FROM maven:3.9.9-eclipse-temurin-21 AS build
WORKDIR /workspace
COPY pom.xml .
COPY src ./src
RUN mvn -q -DskipTests package

FROM eclipse-temurin:21-jre
WORKDIR /app
RUN useradd -u 10001 spring
USER spring
COPY --from=build /workspace/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### Compose local con healthchecks

```yaml
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: orders
      POSTGRES_USER: app
      POSTGRES_PASSWORD: app
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d orders"]
      interval: 5s
      timeout: 3s
      retries: 10

  redis:
    image: redis:7
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 10

  orders-service:
    build: .
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      SPRING_PROFILES_ACTIVE: docker
      DB_URL: jdbc:postgresql://postgres:5432/orders
      DB_USER: app
      DB_PASSWORD: app
      REDIS_HOST: redis
```

### Prompt al agente

> "Genera `Dockerfile` multi-stage y `docker-compose.yml` para `orders-service` + PostgreSQL + Redis.\n> Usa usuario no root en runtime.\n> Incluye healthchecks y `depends_on` por estado saludable.\n> Verifica con `docker compose up --build` y corrige errores de arranque."

### Requisitos de calidad para contenedores

- imagen final sin toolchain de build.
- proceso no root.
- puertos y variables declarados explicitamente.
- readiness/liveness claros para despliegues.

### Prompt de endurecimiento de despliegue

> "Optimiza imagen para arranque rapido y menor superficie de ataque.\n> Devuelve tamaĂ±o aproximado de imagen antes/despues y cambios aplicados.\n> No sacrifiques legibilidad ni reproducibilidad."

### Integracion con tests de integracion

Cuando uses Docker Compose para dev local, no acoples CI a Compose si Testcontainers ya cubre el caso de integracion.

Regla practica:

- local: Compose para levantar dependencias.
- CI: Testcontainers para pruebas autocontenidas.

### Anti-patrones de contenedorizacion

- usar imagen base obsoleta.
- ejecutar como root "porque funciona".
- depender de orden de arranque sin healthchecks.
- mezclar config de `dev` dentro de imagen de `prod`.

## 6. Gate de calidad para codigo generado por IA

La regla no es "si compila, merge". La regla es "si pasa el gate tecnico, merge".

### 6.1 Checklist antes de commit

1. **Arquitectura**: no hay imports cruzados invalidos entre capas.
2. **Semantica**: nombres y contratos coherentes con el dominio.
3. **Testing**: unitarios + integracion para lo tocado.
4. **Seguridad**: sin secretos, sin endpoints expuestos, sin logs sensibles.
5. **Rendimiento**: sin N+1 obvio, sin `findAll()` masivo, con paginacion.
6. **Operabilidad**: errores estandarizados, logs utiles, metricas basicas.

### 6.2 Comandos minimos de verificacion

```bash
mvn -q -DskipTests compile
mvn -q test
mvn -q verify
```

### 6.3 Prompt de auto-auditoria

> "Actua como reviewer Senior. Audita los cambios que acabas de proponer y responde en tabla:\n> - bug funcional\n> - riesgo de seguridad\n> - problema de rendimiento\n> - violacion arquitectonica\n> Para cada hallazgo, indica archivo, severidad y fix recomendado."

### Convertir checklist en gate bloqueante/no bloqueante

Clasifica findings:

- Bloqueante:
  - bug funcional critico.
  - regresion de seguridad.
  - violacion de arquitectura base.
  - tests de integracion rotos.
- No bloqueante:
  - naming inconsistente menor.
  - logs mejorables.
  - simplificaciones de codigo sin riesgo.

### Politica de PR asistida por agentes

Incluye en descripcion de PR:

1. Prompt inicial usado.
2. Restricciones aplicadas.
3. Tests ejecutados.
4. Hallazgos detectados en revision humana.
5. Cambios manuales post-agente.

Ejemplo de plantilla:

```markdown
## IA Session Notes
- Tool mode: Agent multi-model
- Model tier: Balanceado
- Prompt: [resumen]
- Constraints: [lista]

## Verification
- [x] mvn -q -DskipTests compile
- [x] mvn -q test
- [x] mvn -q verify

## Human Review Findings
- [Bloqueante] ...
- [No bloqueante] ...
```

### SeĂ±ales de overtrust que debes detener

- "Los tests pasan, no reviso diff".
- "El agente ya lo audito, no hace falta segunda revision".
- "Si compila en local, merge directo".

### Mini check de 5 preguntas antes de aprobar

1. Entiendo cada cambio en logica de negocio.
2. El agente no toco archivos fuera de scope.
3. No relajamos seguridad para que los tests pasen.
4. No degradamos rendimiento por comodidad.
5. El codigo respeta las invariantes del dominio.

## 7. Flujo recomendado de trabajo: spec -> scaffold -> harden -> verify

Este flujo resume el uso profesional de Copilot/Agente en Spring:

1. **Spec**: define contrato (OpenAPI/ADR/reglas de dominio).
2. **Scaffold**: genera estructura y wiring basico.
3. **Harden**: endurece errores, transacciones, seguridad, observabilidad.
4. **Verify**: ejecuta tests, revisa diff, valida checklist.
5. **Iterate**: corrige con prompts precisos hasta cerrar gaps.

### Prompt maestro de extremo a extremo

> "Implementa la feature `Create Order` desde contrato OpenAPI `openapi/orders-api.yaml`.\n> Stack: Java 21, Spring Boot 3.4, Maven.\n> Arquitectura: hexagonal.\n> Requisitos no funcionales:\n> - Manejo de errores con ProblemDetail\n> - Persistencia con JPA evitando N+1\n> - Test de integracion con Testcontainers\n> - Dockerfile multi-stage\n> Restricciones:\n> - Constructor injection\n> - No Lombok para DTOs\n> - No secretos hardcodeados\n> Criterio de exito:\n> - `mvn test` y `mvn verify` en verde\n> - checklist de calidad sin hallazgos criticos."

### Flujo operativo en sesiones reales

#### Fase 1: Spec (tier alto)

Objetivo: reducir ambiguedad antes de generar codigo.

Prompt sugerido:

> "Antes de codificar, revisa OpenAPI y dame 10 preguntas de clarificacion funcional/tecnica. Si una regla de negocio no esta definida, no asumas; marca gap explicitamente."

#### Fase 2: Scaffold (tier balanceado)

Objetivo: generar estructura coherente con arquitectura.

Prompt sugerido:

> "Genera solo estructura de puertos/adaptadores y contratos. No implementes logica aun."

#### Fase 3: Harden (tier balanceado/alto)

Objetivo: robustecer para produccion.

Prompt sugerido:

> "Ahora endurece manejo de errores, validaciones, concurrencia y config de seguridad. Identifica riesgos restantes."

#### Fase 4: Verify (tier balanceado)

Objetivo: comprobar con evidencia.

Prompt sugerido:

> "Ejecuta tests y reporta fallos con causa raiz y fix aplicado. No modifiques tests para ocultar defectos."

#### Fase 5: Iterate (tier segun necesidad)

Objetivo: cerrar gaps puntuales sin reescribir todo.

Prompt sugerido:

> "Enfocate solo en los findings bloqueantes del diff. No introduzcas cambios colaterales."

### Regla de oro del flujo multi-modelo

No uses siempre el modelo mas potente ni siempre el mas barato.

- si el error tiene coste alto: sube tier.
- si la tarea es mecanica: baja tier.
- si el agente no entiende tras 2 iteraciones: cambia tier o refina especificacion.

### DoD (Definition of Done) para cambios asistidos por agentes

Un cambio se considera listo cuando:

- cumple contrato funcional.
- pasa pruebas automatizadas.
- no introduce deuda arquitectonica evidente.
- tiene observabilidad minima.
- incluye notas de revision humana.

## 8. Plugin del Curso: Asistente de Calidad Spring Boot

En los temas anteriores hemos construido `/refactor-class` y los subagentes de anĂˇlisis e testing. Este tema introduce los primeros dos componentes especĂ­ficos de Spring Boot: un subagente que actĂşa como revisor automĂˇtico de cĂłdigo generado por la IA y una skill que codifica el flujo completo `spec â†’ scaffold â†’ harden â†’ verify`.

La idea es que cuando la IA genere cĂłdigo para una nueva feature de Spring Boot, no tengas que recordar mentalmente cada punto del checklist de este tema. El subagente lo aplica por ti. La skill orquesta el proceso completo para que ninguna fase se salte.

### Subagente: `spring-boot-reviewer`

Este subagente actĂşa como el Senior que revisa cĂłdigo Spring Boot generado por la IA antes de que llegue a un commit. Conoce los anti-patrones especĂ­ficos del tema: N+1, `findAll()` masivo, `@Transactional` mal colocado, secretos hardcodeados, manejo de errores ad-hoc y falta de paginaciĂłn. Es el "gate de calidad" del punto 6 codificado como herramienta reutilizable.

Crea el fichero `.claude/agents/spring-boot-reviewer.md`:

