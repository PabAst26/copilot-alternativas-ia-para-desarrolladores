# Tema 10. MICROSERVICIOS SPRING: PATRONES REALES QUE COPILOT DEBE RESPETAR

En una arquitectura de microservicios, el codigo "aislado" no es suficiente. Aprenderas a usar un agente multi-modelo para implementar resiliencia, consistencia y observabilidad, asegurando que la IA respete la naturaleza distribuida del sistema.

Aunque el titulo mantiene la referencia historica a Copilot, este tema se trabaja con enfoque de plataforma multi-modelo (Copilot Agent, Claude Code u otra alternativa equivalente), aplicando seleccion de tier por tipo de tarea:

- Tier alto: decisiones de arquitectura distribuida, Saga, outbox, contratos criticos.
- Tier balanceado: implementacion multi-servicio, observabilidad, rollout.
- Tier rapido: boilerplate, mappers, tests unitarios simples.

Stack de referencia del tema:

- Java 21+
- Spring Boot 3.4+
- Maven 3.9+
- OpenAPI 3.1
- AsyncAPI 2.x/3.x
- Testcontainers 1.19+
- Micrometer + OpenTelemetry

## 1. Comunicacion Resiliente: REST y Eventos

En microservicios, una llamada remota no es "una llamada de metodo". Es I/O de red con latencia variable, fallos parciales, timeouts y reintentos. Si la IA te genera un cliente HTTP sin politicas de resiliencia, no es un bug menor: es deuda operativa.

### 1.1 Patrones de estabilidad para clientes HTTP

Todo cliente saliente debe declarar al menos:

- timeout de conexion y lectura,
- retry controlado (no infinito),
- circuit breaker,
- fallback o estrategia de degradacion.

**Prompt al Agente:**
> "Implementa un cliente `PaymentClient` con `RestClient` (Spring 6.1+) y Resilience4j. Incluye `CircuitBreaker`, `Retry` con exponential backoff y `TimeLimiter`. Si el proveedor externo falla, responde estado `PENDING_PAYMENT` en vez de lanzar error 500. Ejecuta tests de resiliencia y reporta resultados."

Ejemplo de configuracion `application.yml`:

```yaml
resilience4j:
  circuitbreaker:
    instances:
      paymentClient:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 10
        minimumNumberOfCalls: 5
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3
  retry:
    instances:
      paymentClient:
        maxAttempts: 3
        waitDuration: 200ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
  timelimiter:
    instances:
      paymentClient:
        timeoutDuration: 1500ms
```

Ejemplo de servicio con fallback:

```java
@Service
public class PaymentGatewayService {

    private final ExternalPaymentClient externalPaymentClient;

    public PaymentGatewayService(ExternalPaymentClient externalPaymentClient) {
        this.externalPaymentClient = externalPaymentClient;
    }

    @CircuitBreaker(name = "paymentClient", fallbackMethod = "fallbackAuthorize")
    @Retry(name = "paymentClient")
    public PaymentAuthorization authorize(PaymentCommand command) {
        return externalPaymentClient.authorize(command);
    }

    private PaymentAuthorization fallbackAuthorize(PaymentCommand command, Throwable error) {
        return PaymentAuthorization.pending(command.orderId(), "PAYMENT_PROVIDER_UNAVAILABLE");
    }
}
```

### 1.2 Idempotencia en endpoints de escritura

Si un cliente reintenta `POST /payments`, puedes cobrar dos veces. La idempotencia no es opcional.

Patron base:

1. cliente envia `Idempotency-Key`.
2. servicio revisa si ya proceso esa key.
3. si existe, devuelve misma respuesta previa.
4. si no existe, ejecuta y persiste resultado asociado a la key.

**Prompt al Agente:**
> "Implementa idempotencia para `POST /api/payments` usando header `Idempotency-Key`. Guarda la respuesta en Redis con TTL de 24h y evita doble procesamiento en reintentos concurrentes. Agrega tests para reintento secuencial y reintento concurrente."

Ejemplo de filtro simplificado:

```java
@Component
public class IdempotencyFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain chain)
            throws ServletException, IOException {
        String key = request.getHeader("Idempotency-Key");
        if (key == null || key.isBlank()) {
            chain.doFilter(request, response);
            return;
        }
        chain.doFilter(request, response);
    }
}
```

### 1.3 Mensajeria robusta (Kafka/RabbitMQ)

En eventos, los errores comunes de IA son:

- consumidor sin manejo de retries,
- sin DLQ,
- commits/ACK incorrectos,
- payloads sin version.

**Prompt al Agente:**
> "Genera productor y consumidor Kafka para evento `OrderCreated`. Usa ACK manual, reintentos limitados y DLQ (`order-events-dlq`). Si falla deserializacion o validacion, no pierdas el evento: enrutar a DLQ con metadata de error. Incluye pruebas de integracion con Testcontainers Kafka."

Ejemplo de listener con manejo de error:

```java
@KafkaListener(topics = "order-events", groupId = "inventory-service")
public void onOrderCreated(OrderCreatedEvent event, Acknowledgment ack) {
    inventoryReservationService.reserve(event.orderId(), event.items());
    ack.acknowledge();
}
```

### 1.4 Contratos de eventos

No publiques JSON "ad hoc". Define contrato AsyncAPI.

Ejemplo minimo AsyncAPI:

```yaml

  order/created:
    publish:
      message:

              type: string
              format: uuid
            orderId:
              type: string
              format: uuid

              type: string
```

### 1.5 Checklist de comunicacion resiliente

- timeout definido en todos los clientes salientes.
- retries con maximo de intentos y backoff.
- fallback coherente con negocio (no ocultar errores criticos).
- idempotencia en writes.
- eventos versionados con contrato.
- DLQ operativa y monitorizada.

## 2. Gestion de Contratos y Versionado

La salud de una arquitectura distribuida depende de contratos estables. Si la IA cambia DTOs sin control, rompes consumidores externos aunque tus tests locales sigan verdes.

### 2.1 API First con OpenAPI

OpenAPI 3.1 debe ser la fuente de verdad para REST.

Flujo recomendado:

1. actualizas contrato,
2. revisas breaking changes,
3. implementas adaptadores,
4. validas con contract tests.

**Prompt al Agente:**
> "Toma `openapi/orders-api.yaml` como contrato fuente. Revisa implementacion actual y genera diff de cumplimiento: endpoints faltantes, codigos HTTP inconsistentes y esquemas divergentes. Luego propone cambios minimos para alinear implementacion sin romper backward compatibility."

### 2.2 Detectar breaking changes

Cambios potencialmente incompatibles:

- renombrar campos,
- cambiar tipo (`string` -> `number`),
- marcar required un campo antes opcional,
- eliminar endpoint o cambiar semantica.

**Prompt al Agente:**
> "Analiza este cambio de DTO y clasificalo como `compatible`, `potencialmente incompatible` o `breaking`. Justifica impacto para consumidores Java, JavaScript y mobile."

### 2.3 Estrategias de versionado

Opciones comunes:

- URL (`/v1/orders`),
- Header (`X-API-Version`),
- media type (`application/vnd.orders.v2+json`).

No hay una unica opcion correcta. Debes elegir segun ecosistema y capacidad operativa.

**Prompt al Agente:**
> "Compara versionado por URL vs Header para `orders-service` con 12 consumidores internos y 3 externos. Incluye impacto en gateway, observabilidad, documentacion y migracion progresiva. Recomienda opcion y plan de rollout."

### 2.4 Adaptadores para convivencia v1/v2

En migraciones, evita reescribir dominio dos veces. Usa adaptadores de version.

```java
@RestController
@RequestMapping("/api/v1/orders")
public class OrderControllerV1 {

    private final CreateOrderUseCase createOrderUseCase;

    public OrderControllerV1(CreateOrderUseCase createOrderUseCase) {
        this.createOrderUseCase = createOrderUseCase;
    }

    @PostMapping
    public ResponseEntity<OrderResponseV1> create(@RequestBody @Valid CreateOrderRequestV1 request) {
        CreateOrderCommand command = OrderV1Mapper.toCommand(request);
        Order order = createOrderUseCase.create(command);
        return ResponseEntity.status(HttpStatus.CREATED).body(OrderV1Mapper.toResponse(order));
    }
}
```

### 2.5 Contract testing

No basta con unit tests. Necesitas pruebas de contrato.

Opciones:

- Spring Cloud Contract,
- Pact,
- validacion OpenAPI en CI.

**Prompt al Agente:**
> "Genera contract tests para `GET /api/orders/{id}` y `POST /api/orders` usando OpenAPI 3.1 como referencia. Falla la build si hay divergencia entre contrato e implementacion."

### 2.6 Checklist de contratos y versionado

- contrato OpenAPI/AsyncAPI versionado en repo.
- analisis explicito de compatibilidad en cada PR.
- estrategia de versionado documentada.
- plan de deprecacion con fechas.
- contract tests en CI.

## 3. Configuracion Distribuida y Rollout Seguro

En microservicios, configurar mal equivale a desplegar bugs de forma sincronizada en 10 servicios a la vez.

### 3.1 Configuracion centralizada y refresco controlado

Con Spring Cloud Config (o equivalente), evita valores hardcoded y facilita cambios operativos.

**Prompt al Agente:**
> "Externaliza configuracion de `orders-service` a Config Server. Aplica `@ConfigurationProperties` y usa `@RefreshScope` solo donde sea necesario. No permitas que cambios de config alteren invariantes de dominio en runtime."

Ejemplo de propiedades type-safe:

```java
@ConfigurationProperties(prefix = "orders.limits")
@Validated
public record OrderLimitsProperties(
    @Min(1) @Max(1000) int maxItemsPerOrder,
    @DecimalMin("0.01") BigDecimal minOrderAmount
) {}
```

### 3.2 Feature flags para despliegue progresivo

Feature flags bien usados permiten dark launch y rollback rapido sin redeploy.

Casos tipicos:

- activar nuevo proveedor de pagos al 5%,
- activar nuevo algoritmo solo para tenant beta,
- apagar funcionalidad defectuosa rapidamente.

**Prompt al Agente:**
> "Integra feature flag `orders.new-pricing-engine` con evaluacion por tenant. Si flag esta OFF, usa motor actual. Si ON, usa motor nuevo y emite metrica `pricing_engine_version`. Incluye pruebas para ambos caminos."

Ejemplo de uso simple:

```java
@Service
public class PricingService {

    private final FeatureFlagService featureFlagService;
    private final LegacyPricingEngine legacyPricingEngine;
    private final NewPricingEngine newPricingEngine;

    public PricingService(
            FeatureFlagService featureFlagService,
            LegacyPricingEngine legacyPricingEngine,
            NewPricingEngine newPricingEngine) {
        this.featureFlagService = featureFlagService;
        this.legacyPricingEngine = legacyPricingEngine;
        this.newPricingEngine = newPricingEngine;
    }

    public PriceResult calculate(PricingContext context) {
        if (featureFlagService.isEnabled("orders.new-pricing-engine", context.tenantId())) {
            return newPricingEngine.calculate(context);
        }
        return legacyPricingEngine.calculate(context);
    }
}
```

### 3.3 Rollout seguro (canary y progressive delivery)

Recomendacion operativa:

1. deploy a entorno controlado,
2. canary 1-5%,
3. monitorizacion de SLO/error budget,
4. escalado progresivo,
5. rollback automatizable.

**Prompt al Agente:**
> "Genera plan de rollout para `orders-service` con canary. Define metricas de aceptacion (`error_rate`, `p95_latency`, `payment_timeout_rate`) y umbrales de rollback automatico."

### 3.4 Politicas de configuracion que no se delegan ciegamente

No delegar sin supervision:

- politicas de seguridad y cifrado,
- limites de rate limiting globales,
- estrategias de fail-open/fail-closed,
- configuracion de retencion de datos.

### 3.5 Checklist de configuracion distribuida

- configuracion externa, versionada y auditable.
- `@ConfigurationProperties` validado.
- feature flags con owners y fecha de retiro.
- rollout definido con metricas y umbrales.
- plan de rollback probado.

## 4. Observabilidad: El rastro del microservicio

Si no puedes observar un sistema distribuido, no puedes operarlo. La IA debe generar codigo observable por defecto, no como parche posterior.

### 4.1 Logs estructurados

Minimo en produccion:

- formato JSON,
- `service.name`, `traceId`, `spanId`, `correlationId`,
- campos de negocio utiles (`orderId`, `tenantId`),
- redaccion de datos sensibles.

Ejemplo de evento de log esperado:

```json
{
  "timestamp": "2026-04-12T10:15:23.100Z",
  "level": "INFO",
  "service": "orders-service",
  "traceId": "f68a7b5ea4e5b5c3",
  "spanId": "9d75c32a1b88f01e",
  "correlationId": "8af7f4a6-f75d-45a6-b0cc-219b8a9f2f5c",
  "message": "Order created",
  "orderId": "5ddf4b80-3d71-4d31-a567-b0860ea4e663"
}
```

**Prompt al Agente:**
> "Configura logging JSON para `orders-service` y agrega filtros para propagar `X-Correlation-Id`. Enmascara campos sensibles (`token`, `cardNumber`, `password`) en logs."

### 4.2 Metricas con Micrometer

Tipos minimos:

- contador de operaciones (`orders.created.count`),
- timers (`orders.create.latency`),
- errores por tipo (`orders.create.error.count`),
- metrica de DLQ.

Ejemplo de instrumentacion:

```java
@Service
public class OrderApplicationService {

    private final MeterRegistry meterRegistry;

    public OrderApplicationService(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
    }

    public Order create(CreateOrderCommand command) {
        Timer.Sample sample = Timer.start(meterRegistry);
        try {
            Order order = doCreate(command);
            meterRegistry.counter("orders.created.count").increment();
            return order;
        } catch (RuntimeException ex) {
            meterRegistry.counter("orders.create.error.count", "exception", ex.getClass().getSimpleName())
                .increment();
            throw ex;
        } finally {
            sample.stop(Timer.builder("orders.create.latency")
                .publishPercentileHistogram()
                .register(meterRegistry));
        }
    }

    private Order doCreate(CreateOrderCommand command) {
        return Order.create(command.customerId(), command.items());
    }
}
```

### 4.3 Trazas distribuidas

Con OpenTelemetry + Micrometer Tracing:

- propaga contexto entre llamadas HTTP,
- incluye propagacion en consumidores de eventos,
- correlaciona logs, metricas y trazas.

**Prompt al Agente:**
> "Integra trazas distribuidas en `orders-service` y `payments-service`. Verifica que `traceId` se mantiene en llamada REST y en evento Kafka `OrderCreated`. Incluye test de propagacion basico."

### 4.4 SLO, alertas y runbooks

La observabilidad no acaba en instrumentar.

Define:

- SLO (ej. 99.9% disponibilidad endpoint create order),
- alertas accionables,
- runbook por alerta critica.

Ejemplo de SLO operativo:

- SLI: porcentaje de `POST /orders` con 2xx o 4xx controlado < 300 ms.
- SLO: 99.5% mensual.
- alerta: degradacion > 1% en ventana 10 min.

### 4.5 Checklist de observabilidad distribuida

- logs estructurados y trazables.
- metricas de negocio y tecnicas.
- trazas end-to-end.
- alertas vinculadas a SLO.
- runbooks disponibles para incidencias.

## 5. Concurrencia y Consistencia Distribuida

El error clasico: asumir que `@Transactional` resuelve consistencia entre microservicios. No lo hace.

### 5.1 Transaccion local vs transaccion distribuida

`@Transactional` solo garantiza atomicidad local en una base de datos.

Cuando una operacion implica:

- DB local,
- llamada externa,
- publicacion de evento,

debes diseñar consistencia eventual.

**Prompt al Agente:**
> "Explica por que `@Transactional` no cubre consistencia entre `orders-service` y `payments-service`. Propone diseño con outbox + saga y detalla compensaciones."

### 5.2 Patron Outbox

Objetivo: publicar eventos de forma fiable sin perderlos cuando hay fallo entre DB y broker.

Flujo:

1. guardas entidad de negocio,
2. guardas evento en tabla outbox en la misma transaccion,
3. proceso publicador lee outbox y envia al broker,
4. marcas evento como publicado.

Ejemplo de entidad outbox:

```java
@Entity
@Table(name = "outbox_event")
public class OutboxEventEntity {

    @Id
    private UUID id;

    @Column(nullable = false)
    private String aggregateType;

    @Column(nullable = false)
    private String aggregateId;

    @Column(nullable = false)
    private String eventType;

    @Column(nullable = false, columnDefinition = "text")
    private String payload;

    @Column(nullable = false)
    private Instant createdAt;

    @Column(nullable = false)
    private boolean published;
}
```

**Prompt al Agente:**
> "Implementa patron Outbox para `OrderCreated`. Incluye migracion SQL, repositorio, publisher programado y mecanismo idempotente de marcado `published=true`. Agrega test de integracion que simule fallo temporal del broker y posterior recuperacion."

### 5.3 Saga Pattern (orquestacion o coreografia)

Usa Saga cuando una operacion cruza multiples servicios con pasos compensables.

- Orquestacion: un coordinador central decide pasos.
- Coreografia: cada servicio reacciona a eventos.

No uses Saga para todo. Solo cuando hay proceso distribuido real y compensaciones claras.

Ejemplo simplificado de estados:

- `ORDER_CREATED`
- `PAYMENT_AUTHORIZED`
- `STOCK_RESERVED`
- `ORDER_CONFIRMED`
- compensacion: `PAYMENT_REVERSED`, `STOCK_RELEASED`, `ORDER_CANCELLED`

**Prompt al Agente:**
> "Diseña una saga de `CreateOrder` entre `orders`, `payments`, `inventory`. Define eventos, estados, timeouts y acciones de compensacion. Entrega diagrama textual y pseudocodigo de orquestador en Spring Boot."

### 5.4 Idempotencia en consumidores

En sistemas event-driven, la regla es "al menos una vez". Debes tolerar duplicados.

Patron:

- `processed_event` table con `eventId` unico,
- verificar antes de aplicar side effects,
- si ya existe, descartar duplicado.

```java
@Transactional
public void handle(OrderCreatedEvent event) {
    if (processedEventRepository.existsById(event.eventId())) {
        return;
    }
    reserveStock(event);
    processedEventRepository.save(new ProcessedEvent(event.eventId(), Instant.now()));
}
```

### 5.5 Pruebas clave de consistencia distribuida

- test de outbox con caida de broker.
- test de saga con fallo intermedio y compensacion.
- test de consumidor idempotente ante evento duplicado.
- test de timeout y expiracion de pasos.

### 5.6 Checklist de consistencia distribuida

- no se asume transaccion distribuida magica.
- outbox para eventos criticos.
- saga cuando hay pasos compensables.
- consumidores idempotentes.
- pruebas de fallo y recuperacion.

## 6. Plugin del Curso: Asistente de Resiliencia para Microservicios

Los cinco puntos de este tema definen un estándar claro: ningún cliente HTTP sin circuit breaker, ningún consumidor Kafka sin DLQ, ningún endpoint de escritura sin idempotencia, ningún evento publicado sin outbox. El problema es que recordar ese estándar en cada revisión de código es costoso. Este punto lo convierte en herramienta.

El subagente actúa como un revisor especializado que detecta los anti-patrones de resiliencia antes del commit. La skill orquesta el proceso de añadir resiliencia a un cliente existente siguiendo exactamente el flujo del punto 1, sin improvisar.

### Subagente: `spring-resilience-reviewer`

Revisa código de microservicios Spring Boot en busca de los problemas de resiliencia y consistencia distribuida cubiertos en este tema. No reescribe código — produce un informe de hallazgos que el desarrollador aplica con criterio propio.

Crea el fichero `.claude/agents/spring-resilience-reviewer.md`:

```markdown
---
name: spring-resilience-reviewer
description: Reviewer for microservice resilience patterns in Spring Boot 3.x. Use when auditing HTTP clients, Kafka producers/consumers, or cross-service operations before committing. Detects missing circuit breakers, timeouts, retries, fallbacks, idempotency controls, DLQ configuration, and unsafe distributed consistency patterns (e.g. @Transactional spanning HTTP calls, missing outbox). Does NOT rewrite code — produces a structured findings report only.
model: claude-sonnet-4-6
---

Eres un revisor senior especializado en resiliencia y consistencia distribuida para microservicios Spring Boot 3.4+ con Resilience4j y Apache Kafka.

## Categorías de revisión

Para cada hallazgo indica: archivo afectado, línea aproximada, descripción del problema, severidad y corrección recomendada.

### 1. Clientes HTTP salientes
Para cada clase que use `RestClient`, `WebClient` o `RestTemplate`:
- Ausencia de timeout de conexión y lectura → cualquier llamada puede bloquearse indefinidamente.
- Ausencia de `@CircuitBreaker` (Resilience4j) → un proveedor caído tumba el servicio completo.
- Ausencia de `@Retry` con backoff exponencial → reintentos inmediatos amplifican la carga en el proveedor degradado.
- Ausencia de `@TimeLimiter` → el circuit breaker no puede actuar si la llamada no tiene timeout propio.
- Ausencia de método `fallbackMethod` → el fallo se propaga como excepción sin degradación controlada.
- Fallback que oculta un error crítico de negocio (ej: fallback silencioso en un cobro de pago) → señalarlo explícitamente como riesgo aunque sea decisión de diseño.

### 2. Endpoints de escritura (idempotencia)
Para cada `@PostMapping`, `@PutMapping`, `@PatchMapping`:
- Ausencia de soporte para `Idempotency-Key` en operaciones que modifican estado y pueden ser reintentadas.
- Ausencia de almacenamiento de respuesta previa para evitar doble procesamiento.
- Ausencia de manejo de reintento concurrente (dos requests con la misma key llegando al mismo tiempo).

### 3. Productores y consumidores Kafka
Para cada `@KafkaListener`:
- `AckMode` no configurado como `MANUAL` o `MANUAL_IMMEDIATE` → commits automáticos pueden perder mensajes o procesar duplicados.
- Ausencia de DLQ configurada → mensajes no procesables desaparecen silenciosamente.
- Ausencia de lógica de idempotencia en el consumidor (verificación de `eventId` ya procesado).
- Deserialización sin manejo de error → un mensaje malformado bloquea toda la partición.

Para cada publicación de evento (producer):
- Publicación directa al broker sin patrón outbox → si la BD persiste pero el broker falla, el evento se pierde.
- Ausencia de `eventId` único en el payload del evento → los consumidores no pueden detectar duplicados.

### 4. Consistencia distribuida
- `@Transactional` que engloba una llamada HTTP o una publicación Kafka → la transacción local no garantiza atomicidad con operaciones externas.
- Saga sin compensaciones definidas → si un paso falla, no hay rollback posible.
- Consumidor que aplica side effects antes de confirmar idempotencia → riesgo de doble efecto.
- `processed_event` sin índice único en `event_id` → la protección de idempotencia no es atómica bajo concurrencia.

### 5. Contratos y versionado
- Cambios en campos de DTOs de API sin análisis de compatibilidad → breaking change silencioso.
- Ausencia de `eventId` y `occurredAt` en contratos de eventos → imposible correlacionar ni ordenar.
- Eventos sin versión explícita en el payload → cualquier cambio de esquema es un breaking change.

## Formato de respuesta

1. **Resumen ejecutivo** (2-3 líneas): qué se revisó, cuántos hallazgos por severidad.
2. **Tabla de hallazgos**:

| Categoría | Archivo | Problema | Severidad | Corrección |
|---|---|---|---|---|
| Cliente HTTP | PaymentClient.java:23 | Sin @CircuitBreaker ni fallback | 🔴 Bloqueante | Añadir @CircuitBreaker con fallbackMethod |

3. **Veredicto**: ✅ Listo para commit / ⚠️ Revisión recomendada / 🔴 No commitear sin corregir bloqueantes.

## Severidades

- 🔴 **Bloqueante**: puede provocar indisponibilidad en cascada, pérdida de datos o cobros duplicados (ausencia de circuit breaker, publicación sin outbox en flujo financiero, @Transactional con HTTP).
- 🟡 **Importante**: degrada la operabilidad o acumula deuda de consistencia (sin DLQ, sin idempotencia en consumer, sin Retry).
- 🟢 **Sugerencia**: mejora deseable sin riesgo inmediato (añadir versión a evento, mejorar fallback).
```

```markdown
## Fase 1 — Análisis del componente

Lee el componente indicado y produce:
1. **Tipo de componente**: cliente HTTP / productor Kafka / consumidor Kafka.
2. **Operaciones expuestas a fallos externos**: lista de métodos que llaman a sistemas externos.
3. **Comportamiento actual ante fallo**: excepción propagada / sin manejo / fallback existente.
4. **Dependencias de resiliencia presentes en pom.xml**: Resilience4j, spring-kafka, etc. Si faltan, indícalo.

Muestra el análisis y espera confirmación:
**"¿Procedo con los cambios? (sí/no)"**

## Fase 2A — Resiliencia para clientes HTTP

Si el componente es un cliente HTTP, aplica en este orden:

1. **TimeLimiter**: añade configuración en `application.yml` bajo `resilience4j.timelimiter.instances.<nombre>` con `timeoutDuration`.
2. **Retry**: añade configuración con `maxAttempts`, `waitDuration` y `enableExponentialBackoff`.
3. **CircuitBreaker**: añade configuración con `slidingWindowSize`, `failureRateThreshold`, `waitDurationInOpenState`.
4. **Anotaciones**: decora los métodos que llaman al exterior con `@CircuitBreaker`, `@Retry` y `@TimeLimiter` de Resilience4j. El orden de las anotaciones importa: CircuitBreaker > Retry > TimeLimiter.
5. **Fallback**: añade el método `fallback[NombreOriginal](Throwable)` con el comportamiento de degradación indicado por el usuario. Si no hay comportamiento definido, pregunta antes de inventar uno.

Restricción: no uses `spring-cloud-starter-circuitbreaker-resilience4j`. Usa `resilience4j-spring-boot3` directamente.

## Fase 2B — Resiliencia para consumidores Kafka

Si el componente es un consumidor Kafka, aplica en este orden:

1. **AckMode manual**: configura `spring.kafka.listener.ack-mode: manual` o `MANUAL_IMMEDIATE` en `application.yml`.
2. **ACK explícito**: añade el parámetro `Acknowledgment ack` al método listener y llama a `ack.acknowledge()` solo después de procesar con éxito.
3. **DLQ**: configura `DeadLetterPublishingRecoverer` con el topic `<topic-original>-dlq` en la configuración de Kafka.
4. **Idempotencia**: si el componente no tiene verificación de `eventId` ya procesado, añade la lógica usando una tabla `processed_event` (o un `Set` en memoria si es suficiente para el contexto del ejercicio).
5. **Error handler**: configura `DefaultErrorHandler` con `FixedBackOff(200L, 3)` para reintentos limitados antes de enviar a DLQ.

## Fase 3 — Tests de resiliencia

Genera al menos dos tests:

Para clientes HTTP:
- Test que simula fallo del servidor externo (WireMock o `@MockBean`) y verifica que se invoca el fallback.
- Test que simula timeout y verifica que el circuit breaker registra el fallo.

Para consumidores Kafka:
- Test que envía un mensaje válido y verifica ACK + idempotencia (segundo envío del mismo `eventId` no produce side effect).
- Test que envía un mensaje que provoca excepción y verifica que acaba en DLQ tras los reintentos.

Ejecuta `mvn test -pl <módulo>` y reporta resultado.

Muestra el informe al usuario. Corrige los hallazgos 🔴 Bloqueantes antes de terminar.

- Tests generados y resultado.
```

### ✅ Qué hemos añadido al plugin

- **`spring-resilience-reviewer`** (subagente): audita clientes HTTP, consumidores Kafka y operaciones distribuidas en cinco categorías — clientes HTTP, idempotencia, Kafka, consistencia distribuida y contratos. Referencia directa al estándar de resiliencia del punto 1 y al checklist de consistencia del punto 5.
- **`/add-resilience`** (skill): añade la capa de resiliencia completa a un cliente HTTP (Resilience4j) o a un consumidor Kafka (ACK manual, DLQ, idempotencia) sin tocar la lógica de negocio. Incluye tests de resiliencia y auditoría final con el subagente.
